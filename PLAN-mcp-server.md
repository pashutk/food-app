# MCP Server for Food App — Implementation Plan

## Context

The food-app backend is a TypeScript/Express app on port 3000. It has:
- `dishes` table (CRUD)
- `menus` table (get/set by date)
- JWT Bearer auth
- A cleaned-up service layer for dishes (`src/services/dishes.ts`)
- No service layer for menus (route talks to `db` directly)

The goal is an MCP server that:
1. Runs **alongside** the Express app (same process, same port)
2. Exposes the same internal APIs the app uses
3. Is callable from any MCP-compatible agent
4. Ships in small, independently testable deliverables

**Constraint:** Starting from a blank list. All previous work on the MCP server is discarded.

---

## Non-Goals (from previous attempt)

- NOT a separate Node process
- NOT stdio transport (hard to debug, couples to shell)
- NOT importing route file logic directly into the MCP server

---

## Architecture

```
Agent (MCP client)
    │
    ▼
Express app :3000
    ├── /api/*    ← existing REST API (browser frontend)
    └── /mcp/*    ← MCP HTTP endpoint (StreamableHTTP transport)
```

The MCP server is an Express middleware that mounts on `/mcp`. It shares the same SQLite DB, same JWT auth, same service layer. Each tool calls into the service functions directly (not over HTTP internally).

## Deliverables

### D1 — `services/menus.ts` extraction

**Goal:** Menu operations live in a service file, not in the route.

- Create `src/services/menus.ts` with `getMenu(date)` and `setMenu(date, entries)`
- Rewrite `src/routes/menus.ts` to delegate to the service
- Add `requireAuth` guard to both menu functions in the service (service should NOT assume auth is already checked — middleware handles it at the route layer)
- No behavior change, just extraction

**Files touched:**
- `src/services/menus.ts` (new)
- `src/routes/menus.ts` (rewrite to delegate)

**Test:** Existing e2e menu tests still pass.

---

### D2 — Fix `services/dishes.ts` validation gap

**Goal:** Validation lives in the service, not the route.

Currently `routes/dishes.ts` skips validation when calling service functions. The service (`services/dishes.ts`) already has validation for name being required. The route's `POST /` and `PUT /:id` need to catch service exceptions and return 400.

- `routes/dishes.ts`: wrap `createDish` and `updateDish` calls in try/catch, return 400 on validation error
- `deleteDish`: already returns `{ success: false }` for invalid IDs — route should map that to 404

**Files touched:**
- `src/routes/dishes.ts`

**Test:** `importDishes` return shape — service already returns `{ ok: true, imported }` or `{ ok: false, error, duplicates }`. Route already handles this.

### D7 — `src/mcp/tools/dishes.ts` — Dish CRUD tools

**Goal:** Four tools matching service operations.

- `list_dishes` — calls `listDishes()`, optional filters: `tag`, `takeout`
- `create_dish` — calls `createDish(DishData)`, returns created dish
- `update_dish` — calls `updateDish(id, DishData)`, returns updated dish or error
- `delete_dish` — calls `deleteDish(id)`
- `import_dishes` — calls `importDishes(items[])`, handles duplicate response

All require auth. All return structured JSON text or error.
